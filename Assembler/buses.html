Buses de Comunicación en ATmega328P

Aprende sobre los buses de datos, direcciones y control, y sus diferentes clasificaciones en el ATmega328P

Buses en Arquitectura Harvard

Arquitectura Harvard del ATmega328P

  • Buses separados para instrucciones y datos
  • Mayor velocidad que la arquitectura Von Neumann
  • Permite acceso simultáneo a memoria de programa y datos
  • Optimizado para ejecución eficiente de instrucciones

Buses Principales

  • Bus de Datos: Transfiere información (8 bits)
  • Bus de Direcciones: Especifica ubicaciones (16 bits para Flash)
  • Bus de Control: Gestiona operaciones y temporización
La arquitectura Harvard es característica de la mayoría de los microcontroladores AVR

Bus de Datos

Características del Bus de Datos

  • Bus bidireccional de 8 bits
  • Transfiere datos entre CPU, memoria y periféricos
  • Multiplexado en algunos casos para ahorrar pines
  • Puede operar en diferentes modos (simplex, half-duplex, full-duplex)

Ejemplo en Ensamblador

; Transferencia de datos através del bus
lds r16, 0x0060 ; Leer dato de RAM (bus de datos)
sts 0x0060, r16 ; Escribir dato en RAM (bus de datos)
out PORTB, r16 ; Escribir en puerto (bus de datos)
El bus de datos mueve información entre los diferentes componentes del microcontrolador

Bus de Direcciones

Características del Bus de Direcciones

  • Bus unidireccional de 16 bits para memoria de programa
  • Bus de 16 bits para memoria de datos (pero solo 8 bits para E/S)
  • Generado por el CPU o por instrucciones especiales
  • Selecciona ubicaciones específicas en memoria

Espacios de Direcciones

  • Memoria Flash: 0x0000 - 0x3FFF (32KB)
  • SRAM: 0x0100 - 0x08FF (2KB)
  • EEPROM: 0x0000 - 0x03FF (1KB)
  • Registros de E/S: 0x00 - 0x3F
El bus de direcciones permite acceder a 65536 ubicaciones diferentes (16 bits)

Bus de Control

Señales de Control

  • Señales de lectura (RD) y escritura (WR)
  • Selección de memoria (CS - Chip Select)
  • Señales de interrupción (INT)
  • Señales de reloj (CLK)
  • Reset y control de energía

Funciones del Bus de Control

  • Sincroniza las operaciones del sistema
  • Gestiona el acceso a recursos compartidos
  • Controla la dirección del flujo de datos
  • Gestiona interrupciones y excepciones

Ejemplo de Control

; Habilitar puerto como salida (señal de control)
sbi DDRB, 5 ; Configurar PB5 como salida

; Habilitar interrupción (señal de control)
sbi EIMSK, INT0 ; Habilitar INT0
El bus de control gestiona la temporización y coordinación de todas las operaciones

Comunicación Simplex

Características Simplex

  • Comunicación unidireccional
  • Un dispositivo transmite, otro(s) recibe(n)
  • No hay capacidad de respuesta por el mismo canal
  • Simple pero limitado

Ejemplos en ATmega328P

  • Generación de señales PWM
  • Control de actuadores (LEDs, motores)
  • Escritura en displays
  • Configuración de periféricos

Ejemplo en Código

; Ejemplo de comunicación simplex - Transmitir por PWM
sbi DDRB, 1 ; PB1 como salida (OC1A)
ldi r16, (1<<COM1A1) ; Modo no invertido
sts TCCR1A, r16
ldi r16, 128 ; Ciclo de trabajo 50%
sts OCR1AL, r16
La comunicación simplex es ideal cuando solo se necesita transmitir información en una dirección

Comunicación Half-Duplex

Características Half-Duplex

  • Comunicación bidireccional pero no simultánea
  • Mismo medio para transmitir y recibir
  • Requiere protocolo para alternar entre transmisión y recepción
  • Más eficiente que simplex pero más lento que full-duplex

Ejemplos en ATmega328P

  • Comunicación I2C (TWI)
  • Comunicación SPI en modo half-duplex
  • Protocolos personalizados de un solo cable
  • Comunicación con algunos sensores

Ejemplo con I2C

; Configurar I2C (TWI) - Comunicación half-duplex
ldi r16, 0x48 ; Dirección del esclavo
sts TWAR, r16
ldi r16, (1<<TWEN) ; Habilitar TWI
sts TWCR, r16
El modo half-duplex es común en buses compartidos donde múltiples dispositivos se comunican

Comunicación Full-Duplex

Características Full-Duplex

  • Comunicación bidireccional simultánea
  • Canales separados para transmisión y recepción
  • Máximo rendimiento en comunicación
  • Requiere más recursos que half-duplex

Ejemplos en ATmega328P

  • Comunicación UART (con hardware adicional)
  • Comunicación SPI full-duplex
  • Comunicación con memorias y periféricos avanzados

Ejemplo con SPI

; Configurar SPI - Comunicación full-duplex
ldi r16, (1<<MSTR) ; Modo maestro
out SPCR, r16
sbi DDRB, 3 ; MOSI como salida
sbi DDRB, 5 ; SCK como salida
cbi DDRB, 4 ; MISO como entrada
El modo full-duplex ofrece el máximo rendimiento pero requiere más pines y recursos

Comunicación Serial vs Paralela

Comunicación Serial

  • Transmisión bit a bit por un solo canal
  • Menos pines requeridos
  • Mayor distancia posible
  • Velocidad generalmente más lenta que paralela
  • Ejemplos: UART, I2C, SPI, USB

Comunicación Paralela

  • Transmisión de múltiples bits simultáneamente
  • Más pines requeridos
  • Limitada a distancias cortas
  • Mayor velocidad que serial para distancias cortas
  • Ejemplos: Bus de datos interno, comunicación con memorias externas

Comparación

; Ejemplo serial - Transmitir byte por UART
lds r16, UDR0 ; Leer dato recibido (serial)

; Ejemplo paralelo - Escribir en puerto
out PORTB, r16 ; Escribir 8 bits simultáneos (paralelo)
La comunicación serial es más común para comunicación externa, mientras que la paralela se usa internamente

Comunicación Síncrona vs Asíncrona

Comunicación Síncrona

  • Requiere señal de reloj compartida
  • Mayor velocidad de transferencia
  • Mejor para distancias cortas
  • Ejemplos: SPI, I2C, comunicación con periféricos

Comunicación Asíncrona

  • No requiere reloj compartido
  • Usa bits de inicio y parada para sincronización
  • Mejor para distancias largas
  • Ejemplos: UART, comunicación serial

Ejemplo Síncrono (SPI)

; Transmisión síncrona con SPI
out SPDR, r16 ; Iniciar transmisión
espera_spi:
sbis SPSR, SPIF ; Esperar fin de transmisión
rjmp espera_spi
in r17, SPDR ; Leer dato recibido
La comunicación síncrona es más rápida pero requiere un canal adicional para el reloj

Detección de Errores: Paridad

Paridad en Comunicación

  • Método simple para detección de errores
  • Bit adicional que indica si el número de 1s es par o impar
  • Detección de errores de un solo bit
  • No puede corregir errores, solo detectarlos

Paridad Par

  • Bit de paridad se establece para que el total de 1s sea par
  • Ejemplo: Dato 1011001 (4 unos) → Paridad 0
  • Ejemplo: Dato 1011011 (5 unos) → Paridad 1

Paridad Impar

  • Bit de paridad se establece para que el total de 1s sea impar
  • Ejemplo: Dato 1011001 (4 unos) → Paridad 1
  • Ejemplo: Dato 1011011 (5 unos) → Paridad 0

Configuración en UART

; Configurar UART con paridad par
ldi r16, (1<<UPM01) ; Paridad par
sts UCSR0C, r16

; Configurar UART con paridad impar
ldi r16, (1<<UPM01)|(1<<UPM00) ; Paridad impar
sts UCSR0C, r16
La paridad es un método simple pero efectivo para detectar errores en la transmisión de datos
